# 🧪 مجموعة الاختبارات الآلية — موقع نادي القيروان

اختبارات شاملة (Jest + Supertest) تغطي كل النماذج (Models)، المتحكمات (Controllers)
عبر مساراتها الفعلية (Integration)، والـ Middleware، بالإضافة لحالات حدّية واضحة لكل
ميزة، لتعمل كمجموعة اختبارات رجعية (Regression Suite) عند أي تعديل مستقبلي.

**النتيجة الحالية:** 193 اختبار ناجح ضمن 14 ملف، بتغطية كود تتجاوز 95%.

---

## ⚙️ الإعداد لأول مرة

### 1) قاعدة بيانات اختبار منفصلة تماماً عن التطوير/الإنتاج
```sql
CREATE DATABASE qayrawan_club_test CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

> **مهم جداً:** اسم القاعدة *يجب* أن يحتوي الكلمة `_test`. ملف `tests/helpers/globalSetup.js`
> يرفض العمل تماماً إن لم يجد هذا النمط في `DATABASE_URL`، كحماية من تشغيل الاختبارات
> (التي تحذف الجداول وتعيد إنشاءها) بالخطأ فوق بيانات حقيقية.

### 2) إعداد متغيرات بيئة الاختبار
```bash
cp .env.test.example .env.test
```
ثم عدّل `DATABASE_URL` داخل `.env.test` ليطابق بيانات اتصالك المحلية بقاعدة `qayrawan_club_test`.

### 3) تثبيت الحزم (إن لم تكن مثبتة)
```bash
npm install
```

## 🚀 التشغيل

```bash
npm test                # تشغيل كل الاختبارات مرة واحدة
npm run test:watch      # وضع المراقبة (يعيد التشغيل عند أي تعديل بالملفات)
npm run test:coverage   # تشغيل مع تقرير تغطية الكود (coverage/)
npm run test:db:setup   # إعادة بناء جداول قاعدة بيانات الاختبار يدوياً فقط (بدون تشغيل الاختبارات)
```

كل هذه الأوامر مستقلة تماماً عن قاعدة بيانات التطوير (`qayrawan_club`) المستخدمة في
`npm run dev` / `npm start`، ولا تؤثر عليها بأي شكل.

## 🧠 قرارات تصميمية مهمة

### لماذا قاعدة بيانات حقيقية بدل Mock؟
الاختبارات تعمل على قاعدة بيانات MySQL حقيقية (وليست محاكاة لمكتبة `mysql2`)، لأن
الصحة الفعلية لجمل SQL نفسها (الـ JOIN، `GREATEST()`، `ON DUPLICATE KEY UPDATE`،
الأقواس الخلفية لكلمة `groups` المحجوزة، قيود `FOREIGN KEY`) هي بالضبط ما نريد
التأكد منه. اختبار هذا بمحاكاة (mock) كان سيخفي أخطاء حقيقية بدل كشفها.

### البيانات الثابتة مقابل بيانات الاختبار المتغيرة
جدولا `sessions` (الجلسات التسع للموسم) و`settings` (إعدادات النادي العامة) هما
**بنية أساسية ثابتة** تُملأ مرة واحدة فقط عبر `schema.sql` أثناء `globalSetup`،
وليسا جزءاً من بيانات الاختبار المتغيرة. لذلك:
- `seedFixtures.js` (الذي يُستدعى في `beforeEach` لكل اختبار) **لا يحذف** هذين
  الجدولين، فقط يُفرّغ الجداول المتغيرة (`students`, `groups`, إلخ).
- أي اختبار يحتاج التحقق من سلوك "لا توجد جلسات/إعدادات" يستخدم
  **transaction مع rollback تلقائي** (راجع `statsModel.test.js`) بدل حذف فعلي،
  لتجنّب إفساد هذه البيانات لملفات اختبار أخرى تعمل على نفس قاعدة البيانات.

### بيانات ثابتة (Fixtures) لا عشوائية
`tests/fixtures/seedFixtures.js` يبني بيانات محدَّدة ومعروفة مسبقاً (أسماء، نقاط،
مجموعات بفئتين) خلافاً لملف `config/seed.js` الأصلي الذي يولّد بيانات عشوائية.
هذا يجعل التوقعات (assertions) مثل "الطالب الفلاني رقم 1 بمجموعته" مستقرة 100%
بين كل تشغيل، بدل الاعتماد على حظ التوزيع العشوائي.

### الواجهة الأمامية (public/js)
ملفات `guardian.js` و`supervisor.js` تحتوي منطقاً مرتبطاً بالكامل بالمتصفح
(`document`, `navigator.mediaDevices`, `window.open`) مثل البحث الفوري ومسح
الباركود بالكاميرا. هذا غير قابل للاختبار الآلي بمعزل عن DOM/كاميرا حقيقية
بتكلفة معقولة، لذلك التغطية هنا تتم بشكل غير مباشر: نتحقق عبر اختبارات HTTP
أن الصفحة المُرسلة من الخادم تحتوي كل العناصر (`id`, `data-*`) التي يعتمد
عليها ذلك المنطق (راجع `supervisor.panel.test.js`)، بينما منطق الأعمال
الحقيقي (حساب النقاط، الترتيب، تكرار الحضور) كله خادمي ومُختبر بالكامل.

### خلل تم اكتشافه وإصلاحه أثناء كتابة الاختبارات
أثناء اختبار `GET /api/students/:id` بمعرّف غير رقمي (مثل `/api/students/abc`)،
اكتُشف أن الكود الأصلي كان يرجع **500 وصفحة HTML خطأ** بدل **404 و JSON** كما
تتوقعه أي نقطة API. تم إصلاحه في `controllers/guardianController.js` بإضافة
تحقّق `Number.isInteger()` قبل الاستعلام، وأُضيف اختبار يضمن عدم تكرار هذا
الخلل مستقبلاً.

## 🔧 صيانة هذه المجموعة مستقبلاً

- أي تعديل في `config/schema.sql` (عمود جديد، جدول جديد) يجب أن ينعكس في
  `tests/fixtures/seedFixtures.js` إذا كان يؤثر على البيانات الأساسية المستخدمة بالاختبارات.
- لا تحذف أبداً من `sessions` أو `settings` مباشرة عبر `pool.query()` داخل أي اختبار؛
  استخدم نمط transaction + rollback الموضّح أعلاه إن احتجت اختبار حالة "فارغة".
- عند إضافة controller أو route جديد، أضف ملف اختبار integration مطابق في
  `tests/integration/routes/` يغطي على الأقل: المسار الناجح، حالة عدم وجود البيانات،
  وأي تحقق من صحة المدخلات.
